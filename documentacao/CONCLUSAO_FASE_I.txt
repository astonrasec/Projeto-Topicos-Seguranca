## Arquitetura Implementada

### Padrões de Design

**1. Event-Driven Architecture (ClientManager)**
- Sincronização centralizada de estado
- Classes desacopladas via eventos
- FormLauncher subscreve eventos de ClientManager

**2. Thread-Per-Client (Servidor)**
- Cada cliente em thread dedicada
- Permite paralelismo total
- Sincronização via locks para lista compartilhada

**3. Observer Pattern (UI)**
- FormChat usa InvokeRequired/Invoke para thread-safety
- Thread de receção comunica com UI thread de forma segura
- Sem deadlocks ou race conditions

## Aprendizagens e Decisões Técnicas

### 1. Por que ClientManager (Gerenciador de Clientes)?
**Problema:** FormLauncher precisava saber quando clientes conectavam/desconectavam.
**Solução:** Implementar padrão Event-Driven com classe estática singleton.
**Benefício:** Desacoplamento entre componentes, sincronização automática.

### 2. Por que Thread.IsBackground = true?
**Problema:** Threads bloqueavam a aplicação de fechar.
**Solução:** Marcar threads como background para terminar com a app.
**Resultado:** Limpeza automática de recursos.

### 3. Por que Lock no Servidor?
**Problema:** Multiple threads modificavam lista de clientes simultaneamente (race condition).
**Solução:** Usar lock(clientsLock) para proteger acesso à lista.
**Garantia:** Integridade da estrutura de dados.

### 4. Por que InvokeRequired no Cliente?
**Problema:** Thread de receção não pode modificar controls da UI diretamente.
**Solução:** Usar Invoke() para voltar à UI thread.
**Segurança:** Evita InvalidOperationException e garante thread-safety.

## Limitações Conhecidas (Fase I)

1. **Sem Persistência:** Dados não são salvos (perdem-se ao fechar)
2. **Sem Criptografia:** Mensagens em texto simples
3. **Sem Autenticação Real:** Username aceito sem validação
4. **Sem Segurança em Rede:** Sem validação de assinatura/integridade
5. **IP Fixo:** Precisa conhecer IP do servidor (sem descoberta automática)
6. **Tamanho Máximo:** Sem tratamento para mensagens muito longas
